# copy-alter-row-preservation

## Problem Statement

MyLite's DDL smoke currently proves that an empty `ALTER TABLE ... ALGORITHM=COPY`
can add a column and keep table-definition metadata inside the `.mylite` file.
It does not yet prove the more important storage behavior: when MariaDB copies
rows into the altered table, MyLite must preserve row data, rebuild supported
indexes, retain autoincrement state, and keep all durable state in the primary
file.

This slice adds populated copy-ALTER support for the current MyLite row and key
storage subset.

## Scope

This slice will:

- add storage-smoke coverage for populated copy ALTER on a supported MyLite
  table,
- verify fixed rows, BLOB/TEXT rows, supported primary and secondary keys,
  nullable keys, and autoincrement state survive a copy ALTER,
- verify new-column defaults are materialized by MariaDB during copy,
- verify index reads work through rebuilt index roots after the ALTER,
- verify fresh-process reopen sees the altered table, copied rows, rebuilt
  indexes, and autoincrement state,
- keep unexpected `.frm` sidecar detection intact through the compatibility
  harness.

## Non-Goals

- Do not implement in-place ALTER.
- Do not implement crash recovery for partially completed DDL swaps.
- Do not support ALTER operations whose final table shape is outside MyLite's
  current row/key subset.
- Do not change the row, overflow, index, catalog, or allocator page formats.
- Do not add a DDL journal or WAL companion file.

## Proposed Design

The current MyLite handler may already have enough behavior for MariaDB copy
ALTER because MariaDB creates a new temporary table, opens both source and
destination handlers, converts rows into the destination record layout, calls
`ha_write_row()` for each copied row, and swaps table names through handler
rename hooks.

The implementation should first add smoke coverage that forces
`ALGORITHM=COPY` on populated tables. If the current handler fails, fix the
root cause in the narrow MyLite layer involved:

1. `create()` must accept the altered table definition when the final shape is
   supported.
2. `write_row()` must accept MariaDB's converted destination records during the
   copy loop.
3. Temporary table names generated by MariaDB's ALTER path must be parsed and
   stored in the catalog in a way that later rename/delete hooks can find.
4. Final rename/delete operations must preserve the copied definition and rows
   while removing backup/intermediate catalog entries.
5. Rebuilt key metadata must match the altered table on reopen so loaded
   `INDEXPAGE` roots are accepted.

If those paths already work, the slice lands the tests and documentation as the
release gate. It should not invent a parallel MyLite ALTER implementation.

## Affected Subsystems

- MyLite handler DDL routing.
- MyLite row copy/write path.
- MyLite index-root rebuild and reopen validation.
- Storage smoke and compatibility harness evidence.
- Single-file storage architecture docs and roadmap.

## DDL Metadata Routing Impact

`ALTER TABLE ... ALGORITHM=COPY` should remain MariaDB-driven. MyLite must
persist only the post-ALTER table definition and copied rows under the final
catalog name once the ALTER succeeds. Intermediate and backup names used by
MariaDB's copy algorithm must not remain visible after success.

This slice does not make DDL transactional or crash-safe. A later DDL recovery
slice must design and test partial ALTER recovery.

## Single-File And Embedded-Lifecycle Implications

No companion file is introduced. Successful copy ALTER stores table metadata,
rows, BLOB/TEXT payloads, indexes, autoincrement state, allocator metadata, and
free ranges inside the primary `.mylite` file.

Fresh-process reopen must read the altered table through MariaDB discovery and
serve copied rows and indexes without `.frm` sidecars.

## Public API Or File-Format Impact

No public `libmylite` API change. No file-format version bump is expected.

## Binary-Size Impact

Expected size impact is near zero if the handler paths already work; otherwise
it should stay small and limited to any catalog-name or ALTER-copy handling
fix. Post-implementation `MinSizeRel` sizes will be recorded.

## License, Trademark, And Dependency Impact

No new dependency. All changes remain in existing GPL-2.0-only MyLite and
MariaDB-derived source files.

## Acceptance Criteria

- Populated copy ALTER succeeds for the current supported MyLite row/key subset.
- Copied fixed and BLOB/TEXT values are correct after ALTER.
- MariaDB-applied default values for added columns are visible.
- Primary, secondary, nullable, and BLOB/TEXT prefix index reads work after the
  ALTER.
- Autoincrement continues from the copied table's maximum generated value.
- Fresh-process reopen sees the altered table, copied rows, rebuilt indexes,
  and autoincrement state.
- No unexpected durable `.frm` sidecars or external engine files are introduced.
- Docs and roadmap describe supported copy ALTER and remaining crash-recovery
  limits.

## Risks And Unresolved Questions

- MariaDB's intermediate and backup ALTER names may expose path forms that
  MyLite's parser does not currently normalize correctly.
- The final swap may leave backup catalog entries if MariaDB calls rename and
  delete hooks in an order the current bridge did not anticipate.
- Adding columns with default values depends on MariaDB's copy-loop conversion
  and constraint verification before `ha_write_row()`.
- Crash recovery for a failed process during copy ALTER remains unresolved and
  must not be implied by this slice.

## Implementation Result

Implemented.

The existing MyLite handler paths already supported MariaDB's copy-ALTER
sequence for the current row/key subset. The implementation added release-gate
smoke coverage rather than a storage code change:

- same-process populated copy ALTER with fixed fields, nullable secondary key,
  BLOB/TEXT payloads, BLOB/TEXT prefix key, added-column defaults, rebuilt
  index reads, and post-ALTER autoincrement insert;
- persisted populated copy ALTER with fresh-process reopen checks for copied
  rows, rebuilt indexes, nullable index lookup, BLOB/TEXT prefix lookup, and
  autoincrement state.
